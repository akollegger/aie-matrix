# Research: Ghost CLI

**Feature**: `004-ghost-cli`
**Spec**: [spec.md](./spec.md) · **RFC**: [RFC-0003](../../proposals/rfc/0003-ghost-cli.md)

---

## 1. Command framework: `@effect/cli` vs raw `@effect/platform-node`

**Decision**: Use `@effect/cli` alongside `@effect/platform-node`.

`@effect/cli` provides auto-generated `--help`, `--version`, shell completions, and the `Config` environment-resolution system that reads flags → environment variables → `.env` in a single declarative pass. The `Config` layer eliminates `process.env` guards scattered across pre-flight logic. `@effect/platform-node`'s `NodeRuntime.runMain` manages `SIGTERM`, exit codes, and the top-level fiber lifecycle.

The canonical CLI entrypoint: `NodeRuntime.runMain(Command.run(rootCommand)(process.argv))`.

**Alternatives considered**:

| Alternative | Why not chosen |
|-------------|----------------|
| Raw `effect` + `process.argv` parsing | Loses auto-generated `--help`, `--version`, shell completions, and declarative Config; more boilerplate for no benefit |
| `commander` / `yargs` | No Effect integration; requires manual error-to-exit-code mapping and loses typed error channels |

## 2. Terminal UI: Ink

**Decision**: Use `ink` (v5+) with React 18 for interactive REPL mode only.

Ink renders React components to stdout via an alternative reconciler. It is the industry standard for Node.js TUI work in 2025–2026 (used by Claude Code, Gemini CLI, Amp). It works with ESM and Node.js 24. Ink and Effect coexist cleanly: Effect drives all async logic (MCP calls, reconnect loops) via fibers; React/Ink state (`useState`, `useRef`) is kept to UI-only concerns. Data flows from Effect Refs/Queues into Ink components via a thin adapter layer that calls React's `setState` from inside Effect fibers.

The interactive UI is rendered only when `process.stdout.isTTY` is truthy and no subcommand verb is provided. Non-TTY detection (`process.stdout.isTTY === undefined || !process.stdout.isTTY`) at startup gates the Ink render path.

**Alternatives considered**:

| Alternative | Why not chosen |
|-------------|----------------|
| `blessed` / `neo-blessed` | Largely unmaintained, no Effect integration path |
| `enquirer` | Single-prompt only, no multi-panel layout |
| Plain `readline` with ANSI codes | No component model; maintenance burden |

## 3. Bridging `GhostMcpClient` into an Effect Layer

**Decision**: Wrap `GhostMcpClient` in a `Layer.scoped` that connects on acquire and disconnects on release.

```typescript
// Conceptual — not final code
export class GhostClientService extends Context.Tag("aie-matrix/GhostClientService")<
  GhostClientService,
  { callTool: (name: string, args?: Record<string, unknown>) => Effect.Effect<unknown, GhostClientError> }
>() {}

export const GhostClientLayer = Layer.scoped(
  GhostClientService,
  Effect.gen(function* () {
    const config = yield* GhostConfig;  // token + url from @effect/cli Config
    const raw = new GhostMcpClient({ worldApiBaseUrl: config.url, token: config.token });
    yield* Effect.tryPromise({ try: () => raw.connect(), catch: toGhostClientError });
    yield* Effect.addFinalizer(() =>
      Effect.promise(() => raw.disconnect())
    );
    return {
      callTool: (name, args = {}) =>
        Effect.tryPromise({ try: () => raw.callTool(name, args), catch: toGhostClientError }),
    };
  }),
);
```

The `Layer.scoped` pattern is already established in this codebase for services with async lifecycle (see `TranscriptHubLayer` in `server/src/services/TranscriptHubService.ts`). Using it here keeps the ghost connection tied to the runtime's disposal lifecycle — `NodeRuntime.runMain` handles SIGTERM by disposing the runtime, which closes the MCP transport cleanly.

**Alternatives considered**:

| Alternative | Why not chosen |
|-------------|----------------|
| `Layer.succeed` with manual connect/disconnect | Loses automatic cleanup on SIGTERM and exceptions |
| Constructing client outside Effect | Breaks the service boundary; token/url resolution needs to happen inside Effect's `Config` layer |

## 4. Pre-flight as typed Effects

**Decision**: Implement each pre-flight phase as a separate `Effect.Effect` that yields a typed `Data.TaggedError` on failure.

```typescript
// Conceptual error types
class EnvMissingToken extends Data.TaggedError("PreFlight.EnvMissingToken")<{ inRepoRoot: boolean }> {}
class ServerUnreachable extends Data.TaggedError("PreFlight.ServerUnreachable")<{ host: string; port: number; errno: string }> {}
class TokenRejected extends Data.TaggedError("PreFlight.TokenRejected")<{}> {}
class GhostNotFound extends Data.TaggedError("PreFlight.GhostNotFound")<{}> {}
```

Pre-flight is modeled as a sequential `Effect.gen` that short-circuits on the first failure. Diagnostic messages for each error type are generated by a pure `formatDiagnostic(e)` function, keeping message logic separate from the Effect pipeline.

---

## 5. Failure taxonomy implementation

**Decision**: Map failure types to three stances at the boundary layer.

- **Self-healing**: Reconnect via `Effect.retry(Schedule.exponential("1 second").pipe(Schedule.upTo("30 seconds")))`. In interactive mode the status Ref is updated on each retry attempt; in one-shot mode a single retry is attempted.
- **Guided resolution**: `PreFlight.EnvMissingToken`, `PreFlight.TokenRejected`, `PreFlight.GhostNotFound`, `PreFlight.ServerUnreachable` with `ECONNREFUSED` — each maps to a single concrete remediation message via `formatDiagnostic`.
- **Informative blocking**: Network conditions that are neither ECONNREFUSED nor ENOTFOUND (e.g., firewall drop, TLS error) — state what was observed and direct to server logs.

## 6. REPL command parsing

**Decision**: Implement a minimal command parser in the REPL input handler — no full parser library needed.

The vocabulary is six commands with a small fixed set of argument values. A `splitCommand(input: string): ReplCommand` function converts trimmed input to a discriminated union. Unknown input yields a `ReplCommand.Unknown` variant that triggers inline help display.

```typescript
type ReplCommand =
  | { _tag: "whoami" }
  | { _tag: "whereami" }
  | { _tag: "look"; at: "here" | "around" | "n" | "s" | "ne" | "nw" | "se" | "sw" }
  | { _tag: "exits" }
  | { _tag: "go"; toward: "n" | "s" | "ne" | "nw" | "se" | "sw" }
  | { _tag: "help" }
  | { _tag: "exit" }
  | { _tag: "unknown"; raw: string }
```

## 7. Exit codes

**Decision**: Use distinct exit codes for different failure categories to enable shell script discrimination.

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Configuration / pre-flight failure (user-fixable: missing env var, wrong URL format) |
| 2 | Infrastructure failure (server not running, connection refused) |
| 3 | Authentication failure (token rejected, ghost not found) |

`NodeRuntime.runMain` maps `Effect.die` (defects) to exit code 1 by default; the pre-flight error types map to specific codes via a `toExitCode` function called before `Effect.die` or `process.exit`.
